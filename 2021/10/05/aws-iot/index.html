AWS IOT

Posted on 2021-10-05 In AWS Certification , AWS CCP , AWS SAA Symbols count in article: 4.8k Reading time ≈ 4 mins.

API

An application programming interface (API) defines how software components will interact with one another.

SDK

A software development kit (SDK) is a collection of resources used to help developers extend and integrate a third-party software with their own. It usually leverages the APIs that are provided by the third-party software.

AWS CLI

The AWS Command Line Interface (CLI) is a python-based command line tool used to easily invoke AWS APIs. With the CLI you can describe, create, modify and remove AWS resources.

Signature Version 4

Signature Version 4 is the process to add authentication information to AWS requests. For security, most requests to AWS must be signed with an access key, which consists of an access key ID and secret access key.

Device Gateway

The AWS IoT Device Gateway enables devices to securely and efficiently communicate with AWS IoT. Select here to learn more about the protocols supported by AWS IoT. Things can communicate with each other via the Device Gateway, even if they are using different protocols.

Rules Engine

The rules engine evaluates inbound messages published into AWS IoT, transforms and delivers them to another thing or a cloud based on business rules you define. The rules engine can also route messages to cloud endpoints such as AWS Lambda functions, or a DynamoDB table.

AWS IoT Rules

Rules give your devices the ability to interact with AWS services. Rules are analyzed and actions are performed based on the MQTT topic stream. You can use rules to support tasks like these:

  • Augment or filter data received from a device.
  • Write data received from a device to an Amazon DynamoDB database.
  • Save a file to Amazon S3.
  • Send a push notification to all users using Amazon SNS.
  • Publish data to an Amazon SQS queue.
  • Invoke a Lambda function to extract data.
  • Process messages from a large number of devices using Amazon Kinesis.
  • Send data to the Amazon Elasticsearch Service.
  • Capture a CloudWatch metric.
  • Change a CloudWatch alarm.
  • Send the data from an MQTT message to Amazon Machine Learning to make predictions based on an Amazon ML model.

Before AWS IoT can perform these actions, you must grant it permission to access your AWS resources on your behalf.

X.509 Certificates

X.509 certificates are digital certificates that use the X.509 public key infrastructure standard to associate a public key with an identity contained in a certificate. X.509 certificates are issued by a trusted entity called a certificate authority (CA). The CA maintains one or more special certificates called CA certificates that it uses to issue X.509 certificates. Only the certificate authority has access to CA certificates.

To use AWS IoT certificates, clients must support all of the following in their TLS implementation:

  • TLS 1.2
  • SHA-256 RSA certificate signature validation
  • One of the cipher suites from the TLS cipher suite support section

Thing (Device) Shadow

A thing shadow (sometimes referred to as a device shadow) is a JSON document that is used to store and retrieve current state information for a thing (device, app, and so on). The Thing Shadows service maintains a thing shadow for each thing you connect to AWS IoT. You can use thing shadows to get and set the state of a thing over MQTT or HTTP, regardless of whether the thing is connected to the Internet. Each thing shadow is uniquely identified by its name.

Thing (Device) Registry

The Thing Registry allows you to register your units (both real devices and virtual applications) with the service and associate up to three custom attributes with each unit. Authentication certificates used by these things can also be associated to the thing, to allow for easy viewing and diagnosis of customers IoT fleets.

MQTT

A machine-to-machine/IoT protocol to connect devices to each other and to the cloud. It is a very light-weight protocol designed for resource-limited devices that follows a publish/subscribe messaging format.

WebSockets

A TCP protocol that has been designed for web applications running on web servers or in the browser. While WebSockets leverage HTTP protocol to perform the initial handshake, it is an independent protocol that is meant to keep a connection open for long periods of time while data streams across the connection.

MQTT Client ID

The message broker uses the client ID to identify each client. The client ID is passed in from the client to the message broker as part of the MQTT payload. Two clients with the same client ID are not allowed to be connected concurrently to the message broker. When a client connects to the message broker using a client ID that another client is using, a CONNACK message will be sent to both clients and the currently connected client will be disconnected.

Message Broker

The message broker supports the use of the MQTT protocol to publish and subscribe and the HTTPS protocol to publish. Both protocols are supported through IP version 4 and IP version 6. The message broker also supports MQTT over the WebSocket protocol.
